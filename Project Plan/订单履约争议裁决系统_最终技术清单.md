# 订单履约争议裁决系统：最终技术清单（Docker 本地 MVP 版）

## 1. 项目技术定位

本项目定位为：**争议裁决驱动的人审门控订单履约协作系统**。

第一版目标不是堆满所有企业级技术，而是先在本地通过 Docker Compose 拉起完整链路，验证以下核心能力：

```text
用户发起争议
→ 系统识别纠纷
→ 构建证据卷宗
→ C 层争议审理 Agent Workflow
→ 生成裁决草案
→ 生成执行方案
→ 平台审核员 Human-in-the-loop 确认
→ Tool API 模拟退款、补发、驳回、关闭售后等动作
→ Langfuse 记录全链路 Trace
```

整体技术原则：

```text
Java 负责确定性业务与全局流程
Python 负责 Agent、LLM、RAG、OCR、裁决草案生成
Docker Compose 负责第一版本地部署
Langfuse 负责 Agent Trace 与评估沉淀
Temporal 负责长流程、补证等待、人审等待和超时恢复
```

---

## 2. 最终技术栈总览

| 层级 | 最终选择 | 作用 |
|---|---|---|
| 本地部署 | Docker Compose | 第一版本地一键拉起多服务 |
| 前端 | Vue3 + Vite + Element Plus | 用户补证页、商家补证页、平台审核台 |
| 业务后端 | Java 21 + Spring Boot 3 | 订单、售后、证据、审批、执行、审计 |
| 全局 Workflow | Temporal + Java SDK | Case 生命周期、补证等待、人审等待、超时恢复 |
| Agent 服务 | Python FastAPI + LangGraph | C 层争议审理 Agent Workflow |
| Java ↔ Python 通信 | REST API | 第一版简单稳定，方便调试 |
| 模型网关 | LiteLLM Proxy | 统一接入大模型，避免绑定单一模型 |
| LLM Trace | Langfuse | 记录 Prompt、工具调用、Agent 输出、裁决草案 |
| 数据库 | PostgreSQL | 业务数据、审计数据、Temporal/Langfuse/LiteLLM 周边存储 |
| 缓存 | Redis | 会话缓存、任务状态缓存、限流、短期状态 |
| 文件存储 | MinIO | 存储截图、照片、PDF、质检文件、物流证明等证据 |
| 检索 | Elasticsearch | 政策、证据文本、历史 case 的关键词检索与过滤 |
| OCR | PaddleOCR | 中文截图、物流证明、聊天截图、质检图片识别 |
| 文件转文本 | MarkItDown | 将 PDF、Word、Excel、图片等材料转为 LLM 可读文本 |
| 审批规则 | Spring Boot 内置规则服务 + 配置表 | 第一版实现审批门控，不引入复杂规则引擎 |
| 工具层 | Spring Boot Tool API | 封装订单、物流、证据、政策、消息、执行工具 |
| 反向代理 | Nginx | 统一代理前端、后端、Agent、Langfuse、LiteLLM |

## 3. 最终服务拆分

第一版 Docker Compose 建议拆成以下服务：

```text
frontend
java-api-service
temporal-server
python-agent-service
litellm-proxy
langfuse
postgresql
redis
elasticsearch
minio
ocr-parser-service
nginx
```

### 3.1 frontend

**技术选择：Vue3 + Vite + Element Plus**

职责：

```text
用户补证页面
商家补证页面
平台审核员审核台
Case 时间线展示
证据卷宗展示
裁决草案展示
执行方案确认
```

选择原因：

```text
开发速度快
组件生态成熟
后台管理页面实现成本低
适合快速构建审核台和证据展示页面
```

---

### 3.2 java-api-service

**技术选择：Java 21 + Spring Boot 3**

职责：

```text
订单 case 管理
售后 case 管理
证据卷宗管理
用户/商家补证任务管理
平台审核任务管理
审批规则判断
Tool API 暴露
执行动作模拟
审计记录落库
与 Temporal Workflow 交互
与 Python Agent Service 交互
```

选择原因：

```text
订单、售后、审批、执行、审计属于强业务逻辑
Java/Spring Boot 更适合作为主控业务后端
便于实现权限、事务、幂等、审计和后台 API
与 Temporal Java SDK 集成自然
```

---

### 3.3 temporal-server

**技术选择：Temporal + Java SDK**

职责：

```text
管理全局 Case Workflow
等待用户补证
等待商家补证
等待平台审核员确认
处理补证超时
处理任务重试
保存长流程执行状态
恢复中断流程
```

选择原因：

```text
订单争议流程是长周期流程
用户和商家补证可能需要数小时甚至一天
平台审核也需要 Human-in-the-loop 暂停与恢复
Temporal 比自己手写状态机更可靠
第一版就引入 Temporal，有利于保留企业级架构骨架
```

在本项目中的边界：

```text
Temporal 负责全局流程控制
LangGraph 负责 C 层 Agent 审理逻辑
Tool Executor 负责确定性执行
```

---

### 3.4 python-agent-service

**技术选择：Python FastAPI + LangGraph**

职责：

```text
承载 C 层争议审理 Agent Workflow
提供争点归纳能力
提供证据缺口识别能力
提供补证请求生成能力
提供证据交叉检查能力
提供平台规则适用说明能力
提供裁决草案生成能力
调用 LiteLLM 访问大模型
将 Agent Trace 写入 Langfuse
```

选择原因：

```text
Python 的 LLM、Agent、RAG、OCR、评估生态成熟
LangGraph 适合构建状态化 Agent Workflow
C 层本质是 Agentic Workflow，不适合全部写在 Java 中
FastAPI 适合快速暴露 Agent 服务接口
```

C 层内部结构：

```text
C0 Hearing Controller：由外层 Workflow 控制，不作为自由 Agent
C1 Issue Framing Agent：争点归纳
C2 Evidence Gap & Request Agent：证据缺口识别与补证请求生成
C3 Party Liaison Agent：中立双方联络与材料结构化
C4 Evidence Cross-check Agent：证据一致性与矛盾检测
C5 Rule Application Agent：平台规则适用说明
C6 Adjudication Draft Agent：裁决草案生成
```

---

### 3.5 litellm-proxy

**技术选择：LiteLLM Proxy**

职责：

```text
统一模型调用入口
屏蔽不同模型厂商 API 差异
支持模型切换
支持 fallback
支持调用成本统计
支持后续接入 OpenAI、Anthropic、Gemini、Qwen、DeepSeek 等模型
```

选择原因：

```text
避免 Python Agent Service 与单一模型厂商强绑定
方便后续做模型路由和成本控制
对外提供统一 OpenAI-compatible API
适合本地 Docker Compose 部署
```

---

### 3.6 langfuse

**技术选择：Langfuse**

职责：

```text
记录 Agent 调用链路
记录 Prompt 输入输出
记录工具调用
记录裁决草案生成过程
记录模型调用成本与延迟
沉淀评估数据
支持后续复盘人审修改率和裁决草案质量
```

选择原因：

```text
开源可自托管
适合 Docker Compose 本地部署
相比只打日志，Langfuse 更适合 LLM/Agent Trace
能为后续 Evaluation Agent 提供数据基础
```

---

### 3.7 postgresql

**技术选择：PostgreSQL**

职责：

```text
存储业务 case 数据
存储证据元数据
存储审核记录
存储执行动作记录
存储审计记录
支撑 Temporal 持久化
支撑 Langfuse / LiteLLM 周边数据存储
```

选择原因：

```text
第一版统一用 PostgreSQL 可以减少数据库种类
同时适合业务数据和周边系统持久化
后续可扩展 pgvector，但第一版暂不启用向量检索
```

---

### 3.8 redis

**技术选择：Redis**

职责：

```text
短期会话缓存
补证状态缓存
任务锁
接口限流
临时状态提示
Agent 会话上下文缓存
```

选择原因：

```text
简单成熟
适合作为 Spring Boot 与 Agent 服务之间的短期状态缓存
本地 Docker 部署轻量
```

---

### 3.9 elasticsearch

**技术选择：Elasticsearch**

职责：

```text
政策文本检索
证据文本检索
历史 case 检索
聊天记录检索
质检记录检索
关键词检索
条件过滤
```

选择原因：

```text
订单争议场景大量依赖精确关键词和字段过滤
如订单号、物流单号、SKU、政策编号、序列号、时间范围
第一版不引入向量库，Elasticsearch 已足够支撑政策与证据检索
```

第一版不选向量库的原因：

```text
MVP 重点是跑通争议审理闭环
向量库会增加部署和调试成本
政策和证据检索第一阶段以关键词、字段过滤、全文检索为主
```

---

### 3.10 minio

**技术选择：MinIO**

职责：

```text
存储用户截图
存储商家质检图片
存储物流证明
存储 PDF 政策文件
存储退货照片
存储开箱视频或视频截图
存储 OCR 原始文件
```

选择原因：

```text
本地可模拟 S3 对象存储
适合证据文件管理
Docker 部署简单
后续可平滑迁移到云对象存储
```

---

### 3.11 ocr-parser-service

**技术选择：Python + PaddleOCR + MarkItDown**

职责：

```text
识别聊天截图文字
识别物流证明文字
识别质检图片文字
识别商品序列号
解析 PDF、Word、Excel 等证据材料
将非结构化文件转换为文本
写入 Evidence Store 和 Elasticsearch
```

选择原因：

```text
PaddleOCR 对中文截图、票据、证明类图片更友好
MarkItDown 适合第一版快速把多格式文件转成 LLM 可读文本
Python 生态适合文档解析和 OCR
```

---

### 3.12 nginx

**技术选择：Nginx**

职责：

```text
统一反向代理前端
统一代理 Java API
统一代理 Python Agent API
统一代理 Langfuse
统一代理 LiteLLM
处理本地端口暴露
```

选择原因：

```text
本地部署多服务时访问入口更清晰
前后端分离项目常用
方便后续迁移到 K3s/Kubernetes Ingress
```

```text

        ↓
```

## 6. 第一版暂不采用的技术

第一版为了保证交付效率，明确暂不采用以下技术：

```text
K3s
Kubernetes
Helm
Milvus
Qdrant
MCP
OPA
Drools
Kafka
vLLM
A2A
CrewAI
完整微服务拆分
自部署大模型
```

暂缓原因：

```text
第一版目标是跑通完整业务链路
这些技术会明显增加部署、调试和工程复杂度
当前阶段并非必要
后续可以在 MVP 稳定后逐步增强
```

## 7. 为什么这是第一版最优解

这套技术方案的优势是：

```text
1. 可以用 Docker Compose 本地一键拉起
2. Java 和 Python 分工清晰
3. 保留企业级 Workflow 骨架
4. 保留 Agentic Workflow 核心能力
5. 支持 Human-in-the-loop
6. 支持证据文件、证据文本、政策检索
7. 支持 Agent Trace 和后续评估
8. 不引入第一版不必要的重型技术
```

核心价值：

```text
不是为了展示技术堆叠，而是为了支撑完整的订单争议裁决闭环。
```

## 8. 最终一句话定版

**第一版最终最优技术方案：**

```text
Docker Compose
+ Vue3
+ Spring Boot 3 / Java 21
+ Temporal Java SDK
+ Python FastAPI / LangGraph
+ LiteLLM Proxy
+ Langfuse
+ PostgreSQL
+ Redis
+ MinIO
+ Elasticsearch
+ PaddleOCR
+ MarkItDown
+ Nginx
```

架构定位：

```text
Spring Boot 负责订单、售后、证据、审批、执行、审计等确定性业务；
Temporal 负责任务等待、人审等待、补证等待和全局流程恢复；
Python FastAPI + LangGraph 负责 C 层争议审理 Agent Workflow；
LiteLLM 统一模型调用；
Langfuse 记录 Agent Trace；
PostgreSQL、Redis、MinIO、Elasticsearch 分别承载业务数据、缓存、证据文件和检索能力；
Docker Compose 负责第一版本地完整部署。
```
