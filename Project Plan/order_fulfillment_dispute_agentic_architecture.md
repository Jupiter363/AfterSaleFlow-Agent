# 争议裁决驱动的人审门控订单履约协作系统架构设计

## 0. 文档定位

本文档用于定义一个面向用户与商家的 AI Native 订单履约协作系统架构。系统重点不放在普通订单查询，而放在订单履约过程中最复杂、最有业务价值的部分：**用户与商家之间的履约纠纷、售后争议、证据不完整、责任边界模糊和平台裁决流程**。

本架构采用 **Human-Gated Agentic Workflow** 设计思想：

- **Workflow** 负责流程边界、状态流转、暂停恢复、审批控制、审计记录；
- **Agent** 负责自然语言理解、争点归纳、证据缺口识别、补证请求生成、证据矛盾分析、规则适用说明、裁决草案生成；
- **Human-in-the-loop** 负责高风险节点确认，尤其是平台审核员对裁决草案与执行方案的最终确认；
- **Tool Executor** 负责退款、补发、关闭售后、创建工单、通知双方等确定性执行动作；
- **Evaluation Agent** 负责离线复盘和质量评估，不参与当前 case 的实时裁决。

系统不设计“申诉/复审”流程，所有 case 在平台审核确认并执行后闭环。后续是否支持申诉属于未来扩展，不在当前架构范围内。

## 1. 项目最终定位

### 1.1 项目名称

**争议裁决驱动的人审门控订单履约协作系统**

英文可称为：

**Dispute-Centric Human-Gated Agentic Fulfillment System**

### 1.2 产品定位

本系统不是普通客服机器人，也不是完全自动化售后系统，而是一个围绕订单履约纠纷构建的 AI Native 协作平台。

它的核心目标是：

> 当用户与商家围绕订单履约、物流、签收、退货、退款、补发、商品损坏、掉包、少件、错发等问题产生纠纷时，系统能够自动收集平台客观证据，组织双方补证，归纳争点，分析证据矛盾，适用平台规则，生成裁决草案，并交由平台审核员确认后执行。

### 1.3 核心原则

本架构坚持以下原则：

1. **Agent 不直接当最终法官**  
   Agent 只生成裁决草案，不直接输出最终裁决，也不直接执行退款、补发或关闭售后。

2. **平台审核员是责任锚点**  
   所有涉及退款、补发、驳回、关闭售后、用户权益、商家成本的动作，都需要平台审核员确认。

3. **C 层是受控争议审理 Workflow，不是自由聊天 Agent**  
   C 层内部可以包含多个专用 Agent，但必须由 Hearing Controller 控制流程、轮次、超时、暂停和升级。

4. **B 只做证据卷宗，不做裁决**  
   B 层负责收集、整理、结构化平台证据和双方提交材料，不判断谁对谁错。

5. **D 不重新断案，只映射执行方案**  
   D 层根据 C 的裁决草案或非争议流程的处理结论，生成可执行的 Remedy Plan。

6. **所有路径最终进入平台审核与执行层**  
   无论是否进入争议审理 C，Dispute Router 后的所有路径最终都要进入 D、Approval Policy Engine、Platform Human Review 和 Tool Executor。

## 2. 总体架构

### 2.1 总体架构图

```text
用户 / 商家 / 平台客服 / 平台审核员
        ↓
多渠道入口层
App / Web / 商家后台 / IM / 邮件 / 工单系统
        ↓
A. Case Intake Agent
问题识别、意图理解、售后类型识别、纠纷识别
        ↓
B. Evidence Dossier Builder
平台证据卷宗构建器
订单、物流、支付、售后、聊天、仓储、质检、双方材料
        ↓
Dispute Router
判断进入普通履约流、明确规则流，还是争议审理流
        ↓
┌──────────────────────────────────────────────┐
│ 路径 1：普通履约流                              │
│ Regular Fulfillment Flow                      │
│ 适用于查物流、催发货、普通状态解释等               │
└──────────────────────────────────────────────┘
        ↓
┌──────────────────────────────────────────────┐
│ 路径 2：明确规则流                              │
│ Rule-based Resolution Flow                    │
│ 适用于证据充分、规则明确、无需复杂质证的 case        │
└──────────────────────────────────────────────┘
        ↓
┌──────────────────────────────────────────────┐
│ 路径 3：争议审理流                              │
│ C. Dispute Hearing Workflow                   │
│ 适用于证据冲突、责任不清、用户与商家说法矛盾的 case   │
└──────────────────────────────────────────────┘
        ↓
D. Remedy Planner
将处理结论 / 裁决草案映射成执行方案
        ↓
Approval Policy Engine
审批规则、权限控制、风险分级、审核材料生成
        ↓
Platform Human Review
平台审核员确认 / 修改 / 驳回 / 要求补证 / 转人工
        ↓
Tool Executor
退款、补发、驳回、关闭售后、创建工单、通知双方
        ↓
Case Closure
记录结果、沉淀审计日志、进入离线评估
        ↓
Evaluation Agent
离线复盘、质量评估、案例沉淀、规则缺口发现
```

### 2.2 架构主线说明

系统从入口开始，先由 A 层识别用户或商家的真实诉求，再由 B 层构建证据卷宗。随后 Dispute Router 判断当前 case 是否需要进入 C 层争议审理。

如果不进入 C，case 也不会直接结束，而是进入普通履约流或明确规则流，形成处理结论后，同样进入 D 层生成执行方案，然后经过审批策略、平台审核和执行。

如果进入 C，系统会启动受控的争议审理 Workflow。C 层会围绕争点、证据、规则和双方补证进行有限轮次审理，最后生成裁决草案。裁决草案进入 D 层，生成执行方案，再进入平台审核和执行。

因此，系统的统一后半段是：

```text
处理结论 / 裁决草案
        ↓
D. Remedy Planner
        ↓
Approval Policy Engine
        ↓
Platform Human Review
        ↓
Tool Executor
        ↓
Case Closure
```

## 3. 系统分层设计

### 3.1 入口交互层

入口交互层负责承接不同角色的输入与输出。

| 角色 | 入口 | 主要动作 |
|---|---|---|
| 用户 | App / Web / IM / 邮件 | 发起问题、申请售后、提交证据、确认材料 |
| 商家 | 商家后台 / 工单系统 | 查看争议、提交质检记录、补充发货证明、确认处理意见 |
| 平台客服 | 客服台 | 查看 case、辅助沟通、人工接管 |
| 平台审核员 | 审核台 | 查看裁决草案、确认执行方案、修改或驳回 Agent 建议 |

入口层本身不是智能决策层，它只负责收集输入、展示状态、承载人审动作。

### 3.2 Agentic Workflow 层

Agentic Workflow 层负责整个 case 的流程编排。

核心能力包括：

- case 状态机；
- 普通履约流编排；
- 明确规则流编排；
- 争议审理流编排；
- Agent 节点调用；
- Human-in-the-loop 暂停与恢复；
- 补证轮次控制；
- 补证超时控制；
- 审核节点控制；
- 工具执行节点控制；
- 审计记录落库。

### 3.3 Agent 认知层

Agent 认知层只负责不确定性高、文本复杂、上下文复杂、需要推理和表达的部分。

最终保留的 Agent 包括：

| Agent | 位置 | 核心作用 |
|---|---|---|
| Case Intake Agent | A | 入口意图识别与纠纷识别 |
| Issue Framing Agent | C1 | 争点归纳 |
| Evidence Gap & Request Agent | C2 | 证据缺口识别与补证请求生成 |
| Party Liaison Agent | C3 | 中立联络用户与商家 |
| Evidence Cross-check Agent | C4 | 证据一致性与矛盾检测 |
| Rule Application Agent | C5 | 平台规则适用说明 |
| Adjudication Draft Agent | C6 | 裁决草案生成 |
| Evaluation Agent | 离线 | 裁决质量复盘与案例沉淀 |

### 3.4 确定性工具层

确定性工具层负责调用真实业务系统。

工具类型包括：

| 工具类型 | 示例 |
|---|---|
| 订单工具 | 查询订单状态、支付状态、商品信息、订单金额 |
| 物流工具 | 查询物流轨迹、签收证明、异常节点、物流时效 |
| 售后工具 | 查询售后申请、退货申请、退款状态、售后历史 |
| 仓储工具 | 查询出库记录、发货记录、退货验收、质检记录 |
| 证据工具 | 上传证据、证据归档、元数据提取、文件脱敏 |
| 政策工具 | 查询平台政策、商家政策、政策版本、类目规则 |
| 消息工具 | 通知用户、通知商家、补证提醒、审核结果通知 |
| 审批工具 | 创建审核单、记录审核动作、保存审核意见 |
| 执行工具 | 创建退款单、创建补发单、关闭售后、驳回申请、创建工单 |

### 3.5 数据与审计层

数据与审计层用于保证系统可追溯、可解释、可审核、可复盘。

核心对象包括：

| 对象 | 作用 |
|---|---|
| Fulfillment Case | 履约问题或售后纠纷主对象 |
| Evidence Dossier | 证据卷宗 |
| Claim-Issue-Evidence Matrix | 主张—争点—证据矩阵 |
| Hearing State | C 层争议审理状态 |
| Hearing Record | 庭审过程记录 |
| Adjudication Draft | 裁决草案 |
| Remedy Plan | 执行方案 |
| Review Packet | 平台审核包 |
| Approval Record | 平台审核记录 |
| Action Record | 实际执行动作记录 |
| Evaluation Trace | 离线评估轨迹 |

## 4. 主流程设计

## 4.1 普通履约问题流程

适用于用户只是查询、催促、了解状态，没有明显纠纷的场景。

典型场景：

- 查物流；
- 查发货；
- 查预计送达；
- 普通催发货；
- 普通物流停滞提醒；
- 用户询问售后政策但未发起争议。

流程如下：

```text
用户输入
    ↓
A. Case Intake Agent
识别为普通履约问题
    ↓
B. Evidence Dossier Builder
收集订单、物流、售后基础证据
    ↓
Dispute Router
判断无需进入 C
    ↓
Regular Fulfillment Flow
生成普通处理结论
    ↓
D. Remedy Planner
将处理结论转成通知、催办、创建工单等执行方案
    ↓
Approval Policy Engine
判断是否需要平台审核
    ↓
Platform Human Review
平台审核员确认处理方案
    ↓
Tool Executor
执行通知、催办、创建工单等动作
    ↓
Case Closure
```

注意：即使不进入 C，也仍然进入 D、审批、人审和执行层。这样可以保证所有对用户或商家产生影响的动作都有统一控制面。

---

## 4.2 明确规则问题流程

适用于规则明确、证据充分、无需复杂质证的场景。

典型场景：

- 商家明确同意退款；
- 订单未发货，用户申请取消；
- 明确超时未发货；
- 小额低风险补偿；
- 平台规则明确支持退款或驳回；
- 商品状态、物流状态、售后状态均清晰。

流程如下：

```text
用户或商家发起申请
    ↓
A. Case Intake Agent
识别售后类型与处理诉求
    ↓
B. Evidence Dossier Builder
收集平台证据
    ↓
Dispute Router
判断无需进入 C，进入明确规则流
    ↓
Rule-based Resolution Flow
基于规则生成处理结论
    ↓
D. Remedy Planner
将规则结论映射为执行方案
    ↓
Approval Policy Engine
判断审批路径和风险等级
    ↓
Platform Human Review
平台审核员确认方案
    ↓
Tool Executor
执行退款、驳回、取消、通知等动作
    ↓
Case Closure
```

明确规则流不需要 C 的庭审循环，但仍然保留平台审核员确认节点。

---

## 4.3 争议纠纷流程

适用于证据不完整、责任边界不清、用户和商家说法冲突的场景。

典型场景：

- 用户说未收到货，但物流显示签收；
- 用户申请七天无理由退货，商家称退回商品被掉包；
- 用户称商品收到时已损坏，商家称用户使用后损坏；
- 商家称商品影响二次销售，用户不认可；
- 用户称商家私下承诺退款，但平台记录不完整；
- 少件、错发、漏发责任争议；
- 高价值商品退款争议。

流程如下：

```text
用户或商家发起争议
    ↓
A. Case Intake Agent
识别纠纷类型和双方主张
    ↓
B. Evidence Dossier Builder
构建证据卷宗和争点候选
    ↓
Dispute Router
判断进入 C 争议审理流程
    ↓
C. Dispute Hearing Workflow
争点归纳、补证质证、证据交叉检查、规则适用、裁决草案生成
    ↓
D. Remedy Planner
将裁决草案映射为执行方案
    ↓
Approval Policy Engine
判断审批路径、风险等级和审核材料
    ↓
Platform Human Review
平台审核员确认、修改、驳回或要求补证
    ↓
Tool Executor
执行退款、补发、驳回、关闭售后、通知双方
    ↓
Case Closure
```

## 5. A 层：Case Intake Agent

### 5.1 定位

Case Intake Agent 是入口理解 Agent，负责把用户或商家的自然语言输入转成结构化 case。

它是系统的第一层智能入口，不负责裁决，也不负责生成最终处理方案。

### 5.2 核心职责

- 识别问题类型；
- 识别履约问题、售后问题、争议问题；
- 抽取订单号、售后单号、物流单号；
- 识别用户诉求；
- 识别商家主张；
- 判断是否存在双方说法冲突；
- 判断是否可能进入 C 层争议审理；
- 识别缺失信息并发起补槽。

### 5.3 可调用工具

| 工具 | 用途 |
|---|---|
| 订单基础查询工具 | 确认订单是否存在 |
| 售后单查询工具 | 确认是否已有售后申请 |
| 用户历史 case 查询工具 | 查看用户历史纠纷情况 |
| 商家历史 case 查询工具 | 查看商家历史履约争议情况 |
| 意图分类工具 | 判断用户输入属于哪类问题 |
| 槽位补全工具 | 补充订单号、物流单号、售后原因等 |

### 5.4 输出

A 层输出结构包括：

- case 类型；
- 售后类型；
- 争议类型；
- 用户诉求；
- 商家主张；
- 缺失信息；
- 初步风险等级；
- 是否需要进入 B；
- 是否可能进入 C。

### 5.5 禁止行为

Case Intake Agent 不能：

- 判断谁对谁错；
- 承诺退款；
- 承诺补发；
- 关闭售后；
- 给出最终裁决。

---

## 6. B 层：Evidence Dossier Builder

### 6.1 定位

B 层是平台证据卷宗构建器，可以理解为“平台证据官”。

它不负责裁决，只负责收集、整理、结构化证据。

### 6.2 核心职责

- 收集订单证据；
- 收集支付证据；
- 收集物流证据；
- 收集售后证据；
- 收集平台聊天记录；
- 收集仓储发货记录；
- 收集退货验收记录；
- 收集质检记录；
- 收集用户已提交材料；
- 收集商家已提交材料；
- 构建完整时间线；
- 整理双方主张；
- 生成争点候选；
- 标注证据缺口；
- 生成 Claim-Issue-Evidence Matrix。

### 6.3 可调用工具

| 工具 | 用途 |
|---|---|
| 订单查询工具 | 获取订单状态、金额、商品、商家信息 |
| 支付查询工具 | 获取支付状态、支付金额、退款记录 |
| 物流轨迹工具 | 获取运输、派送、签收等物流节点 |
| 签收证明工具 | 获取签收人、签收时间、签收凭证 |
| 售后记录工具 | 获取售后申请、处理历史、退款记录 |
| 平台聊天记录工具 | 获取平台内用户与商家沟通记录 |
| 商家发货记录工具 | 获取发货时间、发货商品、包裹信息 |
| 仓库质检记录工具 | 获取质检结果、退货验收记录 |
| 退货物流工具 | 获取退货运输与签收记录 |
| 商品类目工具 | 获取商品类目与特殊售后规则候选 |
| 证据上传工具 | 接收用户或商家提交的新材料 |
| 证据脱敏工具 | 对隐私信息进行脱敏处理 |

### 6.4 输出对象

B 层输出以下核心对象：

1. **Evidence Dossier**  
   证据卷宗，包含所有证据、来源、时间、可靠性标注。

2. **Case Timeline**  
   订单和纠纷事件时间线。

3. **Party Claims**  
   用户与商家的主张列表。

4. **Evidence List**  
   已有证据列表。

5. **Missing Evidence**  
   初步缺失证据列表。

6. **Policy Candidates**  
   可能适用的政策候选。

7. **Claim-Issue-Evidence Matrix**  
   主张—争点—证据矩阵。

### 6.5 禁止行为

B 层不能：

- 裁决责任；
- 判断最终结果；
- 生成退款方案；
- 生成补发方案；
- 替用户或商家辩护；
- 选择性隐藏不利证据。

## 7. Dispute Router

### 7.1 定位

Dispute Router 是路由节点，用于判断 case 后续进入哪条处理路径。

它不是最终处理节点，也不是结束节点。

### 7.2 路由目标

Dispute Router 有三类输出路径：

| 路径 | 适用场景 | 后续流向 |
|---|---|---|
| 普通履约流 | 无纠纷、只是查单或催办 | Regular Fulfillment Flow → D → 审批 → 人审 → 执行 |
| 明确规则流 | 规则明确、证据充分 | Rule-based Resolution Flow → D → 审批 → 人审 → 执行 |
| 争议审理流 | 证据冲突、责任不清、双方说法矛盾 | C → D → 审批 → 人审 → 执行 |

### 7.3 路由依据

Dispute Router 主要依据：

- 是否存在双方说法冲突；
- 是否存在关键证据缺口；
- 是否涉及退款、补发、掉包、签收争议等高风险问题；
- 是否有明确平台规则可以直接处理；
- 是否有商家明确同意或用户明确确认；
- 是否需要双方继续补证；
- 是否涉及高价值商品或高风险用户/商家。

### 7.4 关键约束

Dispute Router 不负责裁决，只负责选择流程。

即使不进入 C，也必须进入：

```text
D. Remedy Planner
    ↓
Approval Policy Engine
    ↓
Platform Human Review
    ↓
Tool Executor
```

## 8. C 层：Dispute Hearing Workflow

### 8.1 C 层总定位

C 层是系统核心。

C 不是一个自由裁判 Agent，而是一个受控的争议审理 Workflow。

C 的最终定义是：

> C 是一个平台争议审理工作流，不是单一法官 Agent。它由 Hearing Controller 控制流程，由 Issue Framing Agent 归纳争点，由 Evidence Gap & Request Agent 识别缺证并生成补证请求，由 Party Liaison Agent 中立收集用户和商家材料，由 Evidence Cross-check Agent 检查证据冲突，由 Rule Application Agent 绑定平台规则，最后由 Adjudication Draft Agent 生成带证据链、规则引用、置信度和审核重点的裁决草案。C 只能生成裁决建议和审核包，不能执行退款、补发或关闭售后；所有高风险动作必须进入 D 和平台审核员 human-in-the-loop。

---

## 8.2 C 层内部总体流程

```text
C0. Hearing Controller
开启审理、控制状态、控制轮次、控制暂停与恢复
        ↓
C1. Issue Framing Agent
归纳争点
        ↓
C2. Evidence Gap & Request Agent
识别证据缺口，生成补证请求
        ↓
是否需要补证？
        ├── 是 → C3. Party Liaison Agent
        │           向用户和商家请求补证
        │           ↓
        │        暂停等待双方提交
        │           ↓
        │        B 更新 Evidence Dossier
        │           ↓
        │        C4. Evidence Cross-check Agent
        │           证据一致性与矛盾检测
        │
        └── 否 → C4. Evidence Cross-check Agent
                    直接进入证据交叉检查
        ↓
C0 判断是否继续补证循环
        ↓
C5. Rule Application Agent
平台规则适用
        ↓
C6. Adjudication Draft Agent
生成裁决草案与平台审核包
        ↓
输出给 D. Remedy Planner
```

## 8.3 C0：Hearing Controller

### 定位

Hearing Controller 是 C 层流程控制器，不是 LLM Agent。

它控制争议审理流程的生命周期。

### 核心职责

- 创建争议审理实例；
- 保存 Hearing State；
- 控制 C1 到 C6 的执行顺序；
- 控制补证轮次；
- 控制补证超时；
- 暂停等待用户提交；
- 暂停等待商家提交；
- 暂停等待平台审核；
- 判断是否继续补证循环；
- 判断是否进入人工接管；
- 记录审理过程。

### 可调用工具

| 工具 | 用途 |
|---|---|
| 审理状态读取工具 | 获取当前 hearing 状态 |
| 审理状态保存工具 | 保存当前 hearing 状态 |
| 补证轮次检查工具 | 判断是否达到最大补证轮次 |
| 补证超时检查工具 | 判断用户或商家是否超时未提交 |
| 人审中断工具 | 暂停流程等待用户、商家或平台审核输入 |
| 人工升级工具 | 将 case 转给平台人工处理 |
| 审理记录归档工具 | 保存每个阶段的过程记录 |

### 输出

- 当前审理状态；
- 下一步流程节点；
- 是否继续补证；
- 是否进入 C5/C6；
- 是否升级人工；
- 是否生成审核包。

### 禁止行为

C0 不能：

- 判断责任；
- 判断证据真假；
- 生成裁决；
- 执行退款；
- 执行补发；
- 关闭售后。

## 8.4 C1：Issue Framing Agent

### 定位

Issue Framing Agent 是争点归纳 Agent。

它负责把复杂纠纷拆成平台可以审理的具体争点。

### 核心职责

- 阅读 Evidence Dossier；
- 阅读用户主张；
- 阅读商家主张；
- 识别本案真正争议点；
- 将复杂纠纷拆解为多个可验证问题；
- 为每个争点标注所需证据；
- 判断哪些争点已解决，哪些争点仍待证；
- 形成 Issue List。

### 示例争点

针对“七天无理由退货但商家称用户掉包”的场景，C1 可能归纳出：

1. 用户是否在七天无理由期限内发起退货；
2. 当前商品类目是否支持七天无理由退货；
3. 退回商品是否与发出商品一致；
4. 商家是否提供了足够证据证明退回商品被掉包；
5. 用户是否提供了足够证据证明退回的是原商品；
6. 退回商品是否影响二次销售；
7. 当前证据是否足以支持退款或驳回。

### 可调用工具

| 工具 | 用途 |
|---|---|
| Evidence Dossier 读取工具 | 获取证据卷宗 |
| Case Timeline 读取工具 | 获取时间线 |
| 用户主张读取工具 | 获取用户陈述 |
| 商家主张读取工具 | 获取商家陈述 |
| 政策候选检索工具 | 获取可能相关政策 |
| 相似争议案例检索工具 | 辅助识别常见争点模板 |
| 争议类型模板工具 | 获取不同纠纷类型的争点模板 |

### 输出

- Issue List；
- 每个争点的 required evidence；
- 每个争点的 issue type；
- 每个争点的当前状态；
- 每个争点的优先级。

### 禁止行为

C1 不能：

- 裁决结果；
- 建议退款；
- 建议驳回；
- 向用户或商家发消息；
- 替任何一方组织辩护。

## 8.5 C2：Evidence Gap & Request Agent

### 定位

Evidence Gap & Request Agent 是证据缺口识别与补证请求生成 Agent。

它负责判断每个争点还缺哪些证据，并生成中立、具体、可操作的补证请求。

| 商家私下承诺 | 聊天截图、语音记录、邮件 | 商家客服记录、工单备注 |

### 可调用工具

| 工具 | 用途 |
|---|---|
| Issue List 读取工具 | 获取争点列表 |
| Evidence Matrix 读取工具 | 获取争点和证据关系 |
| 证据要求模板工具 | 获取不同争点所需证据模板 |

## 9. C 层循环控制

### 9.1 C 层循环结构

C 层不是一次性执行，而是一个有限轮次的证据驱动循环。

```text
C1 归纳争点
    ↓
C2 判断证据缺口
    ↓
C3 请求双方补证
    ↓
等待用户 / 商家提交
    ↓
B 更新证据卷宗
    ↓
C4 证据交叉检查
    ↓
C0 判断是否继续补证
    ↓
若继续补证 → 回到 C2 / C3
若证据充分或达到终止条件 → 进入 C5 / C6
```

### 9.2 循环终止条件

C0 根据以下条件终止循环：

1. **关键证据充分**  
   当前证据足以支持生成裁决草案。

2. **达到最大补证轮次**  
   用户和商家已经完成限定轮次补证，仍无法进一步补充。

3. **达到补证超时时间**  
   一方或双方未在规定时间内提交材料。

4. **双方达成一致**  
   用户和商家对解决方案达成一致，不需要继续审理。

5. **出现高风险信号**  
   涉及大额、欺诈、威胁、辱骂、法律风险、平台外交易等情况。

6. **证据仍不可判定**  
   系统无法形成高置信度草案，转平台审核员重点处理。

### 9.3 C 层输出状态

C 层可能输出以下状态：

| 状态 | 说明 | 后续 |
|---|---|---|
| draft_ready | 裁决草案已生成 | 进入 D |
| need_more_evidence | 仍需补证 | 回到 C2/C3 或交平台审核员决定 |
| evidence_insufficient | 证据不足 | 进入 D，生成“证据不足处理方案” |
| manual_required | 需要人工重点审核 | 进入 D，标记高人工优先级 |
| party_agreement | 双方已达成一致 | 进入 D，生成一致方案执行计划 |

---

## 10. D 层：Remedy Planner

### 10.1 定位

D 层是执行方案规划器。

D 不重新判断事实，不重新断案。

D 的作用是：

> 将普通履约结论、明确规则结论或 C 的裁决草案，映射成平台可以审核和执行的 Remedy Plan。

### 10.2 输入来源

D 的输入可能来自三条路径：

| 来源 | 输入内容 |
|---|---|
| 普通履约流 | 普通处理结论，如通知用户、催商家、创建工单 |
| 明确规则流 | 规则处理结论，如支持退款、驳回、取消订单 |
| C 争议审理流 | 裁决草案，如倾向支持用户、支持商家、证据不足、要求补证 |

### 10.3 核心职责

- 读取上游处理结论；
- 判断需要哪些业务动作；
- 生成执行方案；
- 生成用户通知方案；
- 生成商家通知方案；
- 生成平台审核摘要；
- 标注是否涉及退款、补发、驳回、关闭售后；
- 标注执行前置条件。

### 10.4 可规划动作

- 支持用户退款；
- 支持商家拒绝退款；
- 部分退款；
- 补发；
- 换货；
- 要求继续补证；
- 退回商品；
- 关闭售后；
- 创建平台工单；
- 通知用户；
- 通知商家；
- 升级平台人工。

### 10.5 可调用工具

| 工具 | 用途 |
|---|---|
| 裁决草案读取工具 | 读取 C 输出的 Adjudication Draft |
| 处理结论读取工具 | 读取普通流或规则流处理结论 |
| 订单金额读取工具 | 获取退款金额上限 |
| 商品库存读取工具 | 判断是否支持补发 |
| 售后状态读取工具 | 判断当前售后是否可操作 |
| 执行方案生成工具 | 生成 Remedy Plan |
| 通知草案生成工具 | 生成用户和商家通知内容 |
| 审批请求创建工具 | 创建待审核请求 |

### 10.6 禁止行为

D 不能：

- 推翻 C 的事实认定；
- 重新判断谁对谁错；
- 直接执行退款；
- 直接关闭售后；
- 绕过 Approval Policy Engine；
- 绕过 Platform Human Review。

---

## 11. Approval Policy Engine

### 11.1 定位

Approval Policy Engine 是审批规则引擎，不是 Agent。

它负责根据执行方案、金额、风险、角色权限和平台规则，决定需要哪些审核节点。

### 11.2 核心职责

- 判断是否需要平台审核员确认；
- 判断是否需要商家确认；
- 判断是否需要用户确认；
- 判断是否需要风控审核；
- 判断是否需要客服接管；
- 生成审批路径；
- 生成审核材料；
- 标注高风险动作。

### 11.3 典型规则

| 条件 | 审批策略 |
|---|---|
| 退款金额超过阈值 | 平台审核员确认 |
| 补发高价值商品 | 商家确认 + 平台审核 |
| 退货掉包争议 | 平台审核员确认 |
| 高频退款用户 | 风控审核 |
| 证据不足但涉及大额订单 | 平台人工重点审核 |
| 修改地址 | 用户二次确认 |
| 关闭售后 | 平台审核员确认 |

### 11.4 输出

- 审批路径；
- 审批角色；
- 审批原因；
- 审核材料；
- 是否允许进入 Tool Executor。

---

## 12. Platform Human Review

### 12.1 定位

Platform Human Review 是平台审核员最终确认节点。

它是整个系统的责任锚点。

### 12.2 审核员看到的内容

平台审核员应看到完整审核包，包括：

- 订单基础信息；
- 售后基础信息；
- 时间线；
- 用户主张；
- 商家主张；
- 证据卷宗；
- 争点矩阵；
- 证据缺口；
- 证据矛盾；
- 规则适用；
- C 层裁决草案；
- D 层执行方案；
- 风险提示；
- Agent 置信度；
- 建议审核重点。

### 12.3 审核员可执行动作

- 确认裁决草案；
- 修改裁决理由；
- 修改执行方案；
- 要求用户补证；
- 要求商家补证；
- 驳回 Agent 建议；
- 转平台人工客服；
- 转风控或法务；
- 批准进入 Tool Executor。

### 12.4 关键原则

Platform Human Review 不是形式化确认，而是最终责任节点。

Agent 的输出必须是可编辑、可驳回、可追溯的草案。

---

## 13. Tool Executor

### 13.1 定位

Tool Executor 是确定性执行器。

它只执行已经通过审批的动作。

### 13.2 可执行动作

- 创建退款单；
- 创建补发单；
- 创建换货单；
- 驳回售后申请；
- 关闭售后单；
- 创建平台工单；
- 通知用户；
- 通知商家；
- 修改 case 状态；
- 记录执行结果。

### 13.3 禁止行为

Tool Executor 不能：

- 接收未经审批的高风险动作；
- 被 C 层直接触发高风险执行；
- 绕过审计记录；
- 修改 C 层裁决草案；
- 修改平台审核意见。

---

## 14. Evaluation Agent

### 14.1 定位

Evaluation Agent 是离线评估 Agent。

它不参与当前 case 的实时裁决。

### 14.2 核心职责

- 复盘 Agent 裁决草案质量；
- 统计平台审核员修改率；
- 分析证据缺口类型；
- 分析常见争议类型；
- 分析误判风险；
- 沉淀争议案例测试集；
- 发现平台规则缺口；
- 发现流程缺陷；
- 发现 Agent 提示词或策略缺陷。

### 14.3 关键指标

| 指标 | 含义 |
|---|---|
| 裁决草案通过率 | 平台审核员直接通过的比例 |
| 平台审核修改率 | 审核员修改 Agent 草案的比例 |
| 证据缺口识别准确率 | Agent 是否正确识别缺失证据 |
| 补证请求有效率 | 补证请求是否真正帮助解决争点 |
| 争议解决时长 | 从发起纠纷到执行完成的时间 |
| 用户接受率 | 用户对结果的接受程度 |
| 商家接受率 | 商家对结果的接受程度 |
| 高风险误判率 | 涉及退款、掉包、签收争议的严重误判比例 |
| 人工接管率 | 需要平台客服完全接管的比例 |

## 15. Agent 与非 Agent 边界

### 15.1 应该 Agent 化的部分

以下模块适合使用 Agent：

- 自然语言理解；
- 争议识别；
- 争点归纳；
- 证据缺口识别；
- 补证请求生成；
- 双方中立沟通；
- 证据矛盾解释；
- 规则适用说明；
- 裁决草案生成；
- 离线复盘。

这些任务共同特点是：

```text
输入非结构化
上下文复杂
规则难以完全枚举
需要解释和推理
需要生成结构化草案
```

### 15.2 不应该 Agent 化的部分

以下模块不适合使用 Agent：

- 订单查询；
- 物流查询；
- 支付查询；
- 退款执行；
- 补发执行；
- 关闭售后；
- 审批路由；
- 权限校验；
- 状态落库；
- 审计记录；
- 定时任务。

这些任务共同特点是：

```text
输入输出明确
强事务性
强审计要求
失败成本高
需要确定性执行
```

## 18. 典型场景架构流转

### 18.1 场景一：用户查物流

```text
用户问：我的订单到哪了？
    ↓
A 判断为普通履约问题
    ↓
B 收集订单和物流证据
    ↓
Router 不进入 C
    ↓
Regular Fulfillment Flow 生成物流解释
    ↓
D 生成通知方案
    ↓
Approval Policy Engine 标记低风险
    ↓
Platform Human Review 确认
    ↓
Tool Executor 通知用户
```

### 18.2 场景二：明确超时未发货

```text
用户申请退款，订单超过平台发货时限，商家未发货
    ↓
A 判断为售后申请
    ↓
B 收集订单、支付、发货状态
    ↓
Router 判断规则明确，不进入 C
    ↓
Rule-based Resolution Flow 生成支持退款结论
    ↓
D 生成退款执行方案
    ↓
Approval Policy Engine 判断需要平台审核
    ↓
Platform Human Review 确认
    ↓
Tool Executor 创建退款单并通知双方
```

### 18.3 场景三：七天无理由退货但商家称掉包

```text
用户申请七天无理由退货
商家称退回商品被掉包
    ↓
A 判断为退货争议
    ↓
B 收集订单、签收时间、退货申请、退货物流、质检记录、双方材料
    ↓
Router 判断证据冲突，进入 C
    ↓
C1 归纳争点：是否在七天内、商品是否一致、商家掉包证据是否充分
    ↓
C2 识别缺证：发货前序列号、退货开箱视频、用户退货前照片
    ↓
C3 请求用户和商家补证
    ↓
B 更新证据卷宗
    ↓
C4 检查序列号、照片、时间线和质检记录是否一致
    ↓
C5 匹配七天无理由、商品一致性、举证责任规则
    ↓
C6 生成裁决草案
    ↓
D 将裁决草案映射为退款、驳回、继续补证或人工重点审核方案
    ↓
Approval Policy Engine 判断审批路径
    ↓
Platform Human Review 确认
    ↓
Tool Executor 执行最终动作
```

### 18.4 场景四：用户称未收到但物流显示签收

```text
用户称未收到商品
物流显示已签收
    ↓
A 判断为签收争议
    ↓
B 收集物流签收记录、签收证明、用户历史记录、商家发货记录
    ↓
Router 判断存在责任争议，进入 C
    ↓
C1 归纳争点：是否实际签收、签收证明是否充分、用户是否能提供未收到证据
    ↓
C2 生成补证请求：用户提供快递沟通截图，商家提供发货和物流凭证
    ↓
C3 中立收集双方材料
    ↓
C4 校验物流轨迹、签收时间、签收凭证、双方说法
    ↓
C5 适用签收争议规则和举证责任规则
    ↓
C6 生成裁决草案
    ↓
D 生成执行方案
    ↓
平台审核
    ↓
执行
```

## 19. 最终架构总结

本系统的最终架构可以概括为：

> 这是一个以争议裁决为核心的人审门控订单履约协作系统。系统首先通过 Case Intake Agent 识别用户或商家的真实诉求，通过 Evidence Dossier Builder 构建平台证据卷宗，并由 Dispute Router 判断 case 进入普通履约流、明确规则流还是争议审理流。无论是否进入争议审理，所有路径最终都会进入 D. Remedy Planner、Approval Policy Engine、Platform Human Review 和 Tool Executor，保证所有影响用户权益和商家成本的动作都可审核、可追溯、可控制。

其中，C 层是系统核心。C 不是自由裁判 Agent，而是平台争议审理 Workflow。它由 Hearing Controller 控制流程，由 Issue Framing Agent 归纳争点，由 Evidence Gap & Request Agent 识别补证需求，由 Party Liaison Agent 中立收集双方材料，由 Evidence Cross-check Agent 检查证据矛盾，由 Rule Application Agent 适用平台规则，最后由 Adjudication Draft Agent 生成裁决草案。C 只能生成裁决建议和审核包，不能直接退款、补发或关闭售后。

最终执行必须经过平台审核员 human-in-the-loop，由审核员确认、修改或驳回 Agent 生成的裁决草案和执行方案。通过审核后，Tool Executor 才能执行退款、补发、驳回、关闭售后、通知双方等确定性动作。

整个架构的关键价值在于：

- 用 Agent 解决订单纠纷中最难的争点归纳、证据分析、补证推进和裁决草案生成；
- 用 Workflow 控制流程边界、轮次、超时、暂停与恢复；
- 用 Human-in-the-loop 控制高风险裁决和执行；
- 用 Tool Executor 保证真实业务动作确定、可审计、可追责；
- 用 Evaluation Agent 沉淀离线评估能力，持续优化争议处理质量。

这不是一个“LLM 直接判案”的系统，而是一个“平台规则约束下，由 Agent 辅助审理、由人最终确认、由工具确定执行”的 AI Native 订单履约争议裁决架构。
