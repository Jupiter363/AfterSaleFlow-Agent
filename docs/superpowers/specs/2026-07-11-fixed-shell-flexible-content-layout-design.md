# 固定外框与弹性内容区统一布局设计

日期：2026-07-11  
状态：已完成方案确认，待实施计划  
关联审计：`docs/acceptance/frontend-long-text-layout-audit-2026-07-10.md`

## 1. 背景

当前前端已经形成稳定的视觉语言：接待室、证据室和庭审页都采用固定尺寸的房间卡片，状态卡、数字人消息卡和操作台也有明确的最小尺寸。问题不在于“卡片必须固定”，而在于固定外框内部仍混用了固定行高、自动 Grid 行、多层 `overflow: hidden` 和断点后的纵向堆叠，导致内容一旦增长就会互相争夺空间。

真实浏览器审计确认了 17 项布局问题，其中最严重的包括：

- 庭审中央区在约 1024×600 及更窄视口下被压缩到约 28px。
- 接待室在 580px 断点后原始陈述正文高度归零。
- 总览页阶段 4、5 被案件信息卡覆盖。
- 接待室和证据室在 981–1025px 范围仍保留放不下的双列布局。
- Outcome 草案在 390px 下产生页面级横向溢出。
- 长审核包将同排卡片和侧栏拉伸数千像素，形成巨量空白。
- 传票信箱在移动端偏出视口，顶部控制区会随通知列表滚走。

## 2. 已确认约束

1. 已经形成产品认知的外层卡片尺寸保持稳定，不改成任意内容高度。
2. 固定尺寸可以按真实业务文本重新分配内部区域比例。
3. 正常业务文本应尽量完整展示，避免每个小卡片都出现滚动条或展开按钮。
4. 异常长文本才进入局部滚动、摘要折叠或“查看全文”。
5. 短字段应充分利用横向空间，采用多列多行布局；长正文独占整行。
6. 接待室和证据室在窄屏下上下排列，每张主卡仍保持固定高度，页面负责在主卡之间纵向滚动。
7. 庭审页在窄屏下保留完整中央庭审台，左右证据栏改为侧边抽屉，不能将三块固定卡纵向塞入同一庭审画布。
8. 本轮只处理前端布局和展示行为，不改变后端业务合同。

## 3. 方案比较

### 方案 A：固定外框合同 + 单一滚动轨道 + 内容类型网格（采用）

- 外框、头部、输入区和操作区保持稳定。
- 中间内容区使用 `minmax(0, 1fr)`。
- 每个主卡只定义一个主要滚动轨道。
- 短字段多列并排，长字段跨满整行。
- 依据典型值和 P95 业务字数重新分配固定空间。

优点：保持现有产品观感，正常文本下少滚动，异常文本有明确出口。  
代价：需要逐页梳理 Grid 责任和响应式断点。

### 方案 B：所有卡片随内容自然增高（不采用）

实现简单，但会破坏固定工作台视觉，并让长庭审或 ReviewPacket 把页面拉到数千像素。

### 方案 C：所有小卡片固定高度并分别滚动（不采用）

外观稳定，但会形成大量嵌套滚动，普通文本也需要频繁滚动，阅读体验割裂。

## 4. 统一布局合同

### 4.1 固定外框

房间类主卡统一遵循：

```text
固定外层卡片
├─ 固定头部：标题、身份、状态
├─ 弹性内容区：minmax(0, 1fr)
│  └─ 唯一主要滚动轨道
└─ 固定底部：输入台或操作按钮
```

所有参与弹性高度计算的父子层都必须具备：

```css
box-sizing: border-box;
min-width: 0;
min-height: 0;
```

外层卡片可以 `overflow: hidden/clip`，但只能裁剪装饰，不得依赖它裁剪业务正文。
所有固定预算至少保留 8px 用于边框、浏览器舍入和字体渲染误差，不把理论高度刚好用满。

### 4.2 单一滚动责任

每张主卡只保留一个主要内容滚动区：

- 接待室/证据室左卡：消息列表。
- 接待室右卡：正常情况下无主要滚动；异常原始陈述或全文弹层按需滚动。
- 证据室右卡：证据列表。
- 庭审中央卡：庭审消息列表。
- 庭审两侧：各自证据列表。
- Review 卡：仅正文超过业务容量时滚动。
- Mailbox：通知列表。

标题、状态、输入台、操作按钮、抽屉 Header 不参与日常滚动。

### 4.3 文本类型与展示策略

| 类型 | 示例 | 正常策略 | 异常策略 |
|---|---|---|---|
| 短状态 | 风险、完善度、可信分、提交状态 | 横向多列、允许换行 | 不滚动，使用受控短文案 |
| 标识字段 | 案件号、订单号、文件名 | 单行省略，保留完整查看入口 | `title`、详情弹层或复制能力 |
| 中等摘要 | 诉求、回应、案情摘要 | 2–6 行完整预算 | 发生真实溢出时显示“查看全文” |
| 长正文 | 原始陈述、聊天消息、裁决解释 | 独占整行、自然增高 | 超长报告折叠或局部滚动 |
| 列表 | 核验重点、争点、风险提示 | 典型条数完整展示 | “另有 N 项”或正文滚动 |

所有自然语言正文统一使用：

```css
white-space: pre-wrap;
overflow-wrap: anywhere;
word-break: break-word;
```

禁止通过全局 `overflow-x: hidden` 掩盖真实宽度错误。

### 4.4 可复用能力

实施阶段应优先形成三个小型公共能力，而不是复制各页逻辑：

1. `ExpandableText`
   - 接受预览行数。
   - 只有真实发生溢出时才显示“展开/收起”。
   - 使用 `aria-expanded`，支持键盘操作。

2. `StablePanel` 或等价布局类
   - 统一头部、弹性内容区、底部操作槽。
   - 只负责尺寸和滚动责任，不承载业务样式。

3. `MetaGrid`
   - 支持 12 列布局、字段跨列和窄屏规则。
   - 短字段横向排列，长字段跨满整行。

不要求一次性重构所有页面；可先以共享 CSS 合同实现，再在重复明显时抽组件。

## 5. 接待室与证据室

### 5.1 主卡与响应式规则

- 左右主卡继续保持 `740px` 高。
- 容器实际可用宽度 `>= 1060px` 时使用双列，`< 1060px` 时改为上下单列。
- 单列时每张仍为 `740px`，页面负责纵向滚动。
- 容器查询是规范实现；如暂不引入，fallback 明确为 viewport `>= 1120px` 双列、`< 1120px` 单列。

该规则取代当前 980px 断点，解决 981–1025px 范围的页面级横向溢出。

### 5.2 左侧聊天卡

固定预算：

| 区域 | 高度预算 |
|---|---:|
| 案情提示头 | 118–126px |
| 消息列表 | 剩余弹性空间，约 400px 以上 |
| 输入台 | 126–138px |

聊天消息规则：

- 消息卡保留现有最小高度，正文增长时自然增高。
- 普通消息不设置卡内滚动或 line-clamp。
- 消息列表是唯一滚动区。
- textarea 固定约 72px，禁止拖拽改变外框，自身允许滚动。
- 手机端气泡最大宽度提高到 92%–94%。
- 约 1500 字或 8KB 以上的异常报告才进入“展开全文”。

### 5.3 接待室右侧卷宗

`740px` 外框内部采用 `border-box`，预算如下：

| 区域 | 高度预算 |
|---|---:|
| 卷宗标题 | 44px |
| 完善度与风险 | 44px |
| 争议详情主体 | 412px |
| 下一步核验重点 | 112px |
| 底部操作区 | 52px |
| 5 处间距 | 40px |
| 上下内边距 | 28px |
| 边框与舍入余量 | 8px |

争议详情采用 12 列网格：

```text
案情摘要                         12列
发起方诉求          6列 | 对方回应          6列
订单索引      4列 | 售后索引      4列 | 物流索引      4列
原始陈述                         12列
```

内容宽度小于 420px 时仍优先利用横向空间：诉求和回应保持两列，但使用更短的归一化摘要；订单、售后、物流保持三等分。所有完整原文均可从全文入口查看，避免移动端将短字段改成纯纵向堆叠。

争议详情整卡为 412px，其中包含标题、内边距和装饰；真正分配给正文子区域的稳定槽约为 354px：

| 子区域 | 桌面 | `<420px` 内容宽度 |
|---|---:|---:|
| 案情摘要 | 110px，最多 5 行 | 96px，最多 4 行 |
| 诉求与回应 | 70px，两列 | 84px，两列、每项最多 3 行 |
| 三项案件索引 | 42px | 42px |
| 原始陈述 | 108px，最多 4 行 | 108px，最多 4 行 |
| 子区域间距 | 24px | 24px |

窄屏下“正常完整展示”是指归一化摘要完整；发起方原话和超过预览容量的正文通过 `ExpandableText` 或全文弹层访问。移动端不承诺在 740px 内同时无折叠展示 150 字摘要和 160 字原话全文，这是物理空间限制，不应通过压缩行高伪装完成。

典型/P95 文本预算：

- 案情摘要：桌面 80–150 字尽量完整；窄屏预览约 80–110 字，溢出时出现全文入口。
- 发起方诉求、对方回应：桌面 12–40 字；窄屏使用 12–30 字归一化摘要，完整表述进入详情。
- 原始陈述：桌面预留约 100–160 字；窄屏预览约 80–120 字，完整原话通过展开或全文弹层访问。
- 核验重点：正常完整显示 3–4 项，每项允许 2 行；更多项显示“另有 N 项”。

580px 以下底部两个动作按钮仍保持两列和固定高度。若长文案放不下，缩短次操作文案或收入更多菜单，不得通过纵向堆叠挤掉正文空间。

### 5.4 证据室右侧卡

桌面预算：

| 区域 | 高度预算 |
|---|---:|
| 标题与身份 | 70–76px |
| 上传台 | 82–86px |
| 证据列表 | 剩余约 440–460px |
| 状态与操作 | 58–64px |

320px 紧凑预算：

| 区域 | 高度预算 |
|---|---:|
| 标题与身份 | 84–88px |
| 上传台 | 92–96px |
| 证据列表 | 剩余约 395–415px |
| 状态与操作 | 66–72px |

证据列表是唯一主要滚动区。证据库分组自然增高，不再各自嵌套滚动。

`<= 360px` 时房间头部使用明确的多行 Grid：

```text
证据书记官室
CASE_…                    举证窗口 01:59:59
AI 非最终建议提示（独占整行并允许换行）
```

仅倒计时本身允许 `nowrap`；房间标题、案件标识和 AI 提示都必须 `min-width: 0` 并允许换行。数字人开场消息所在的 ConversationStream 全链路同样必须 `min-width: 0`，不得再次横向撑宽页面。

证据卡规则：

- 文件图标 + 文件信息为主布局。
- 文件名单行省略，点击现有详情弹层查看完整值。
- 类型、提交状态、核验状态和可信分位于可换行的短字段行。
- 长证据说明独占整行。
- 320px 下隐藏非业务装饰，并将状态行移到文件信息下方。
- 620px 以下底部仍保持“说明 + 操作按钮”横向 Grid，不通过单列堆叠增加底部高度。

通用错误使用 toast、弹层或不参与 Grid 高度计算的提示层，不得插入固定卡片的新行。

## 6. 庭审页

### 6.1 固定庭审画布

外框继续使用：

```css
height: clamp(720px, calc(100dvh - 150px), 820px);
```

桌面布局：

```text
用户证据栏 282px | 中央庭审台 | 商家证据栏 282px
```

当容器可用宽度 `>= 1220px` 时使用三栏，`< 1220px` 时：

- 中央庭审台保持完整固定画布。
- 左右证据栏改为覆盖式侧边抽屉。
- 同一时间最多打开一侧。
- 不再把三块纵向堆入同一个 720px 外框。

### 6.2 中央区域预算

中央区使用显式三行：

```text
当前庭审状态：122px
庭审消息区：minmax(0, 1fr)
本轮陈述输入台：154px
```

中央区统一 `border-box`。以最小 720px 外框计算：上下 padding 各 14px、两处 gap 各 10px、状态坞 122px、输入坞 154px，因此消息区精确预算为 `720 - 28 - 20 - 122 - 154 = 396px`；820px 外框时为 496px。审核员无输入台时使用明确状态属性，将 154px 返还消息区，不依赖 Grid 自动行。

错误提示不得成为中央 Grid 的额外行。

### 6.3 状态坞

- 保持 122px。
- “当前庭审状态”和倒计时位于第一行。
- 当前阶段短标题独占第二行。
- 三阶段始终横向三列；680px 以下只把每个阶段内部的“名称 + 状态”改为上下排列。
- 长法官说明进入消息区，不进入固定状态卡。

### 6.4 庭审消息

- `.court-transcript__messages` 是唯一日常消息滚动区。
- 消息卡保留最小高度并自然增高，不截断正文。
- 身份和时间位于消息头第一行。
- 风险等级、可信分、复核结论等短字段位于第二行多列区域。
- 用户/商家典型 80–300 字；法官/书记官典型 120–500 字；评审团典型 300–800 字。
- 以 Unicode 字符数为准，正文达到 1500 字符时进入异常长报告模式；不再混用字节阈值。

### 6.5 输入坞

- 保持 154px。
- 内部使用“22px 状态行 + 弹性 composer”两行 Grid。
- 680px 以下仍保持 textarea 与约 104px 按钮列横向排列。
- textarea 自身滚动。
- 正常态、锁定态、最终封存态复用同一个固定内容槽。

### 6.6 证据栏

```text
标题、角色、数量：约 88px
证据列表：minmax(0, 1fr)
底部操作：48px
```

证据栏同样使用 `border-box`：上下 padding 共 28px、两处 gap 共 20px、头部 88px、底部 48px，列表预算约 536px。普通证据卡 `min-height: 78px`、典型高度 78–92px，因此可正常显示约 5–6 张，更多证据进入列表滚动。

## 7. 总览页

总览继续保持固定工作台，不再使用允许回落到失败值 659px 的模糊范围：

| 宽度 | 工作区高度 |
|---|---:|
| `>= 1021px` | `clamp(720px, calc(100dvh - 170px), 780px)` |
| `681–1020px` | `810px` |
| `361–680px` | `880px` |
| `<= 360px` | `940px` |

关键规则：

- 五阶段始终保持横向轨道；中小屏由轨道自身横向滚动。
- 不再将五阶段改为五行。
- 案件详情区保持 2×2，而不是四卡单列。
- 每张索引卡约 105–115px，内容纵向居中。
- 案件号和订单号单行省略；下一步动作允许两行。
- 顶部两个主要操作保持两列，只有 ≤360px 才改单列。

中等宽度使用约 186px 案件轨道 + 18px 间距 + 606px 流程工作区；361–680px 使用约 210px 案件轨道 + 18px 间距 + 652px 流程工作区。≤360px 新增的 60px 用于容纳单列操作按钮。流程/态势区域不得以 `overflow: hidden` 静默裁掉业务内容；其内部行预算必须小于对应固定工作区。

## 8. Outcome

Outcome 是裁决文档，不固定整个页面总高度；页面继续纵向滚动，但各业务模块使用稳定的正常尺寸。

Outcome 各模块使用 `min-height`，不是固定 `height`；正常内容可自然增长，异常长正文才折叠，不给文档流制造普遍嵌套滚动。

| 模块 | `min-height` 建议 |
|---|---:|
| Hero | 桌面约 128px；移动端约 158px |
| 白话结论 | 正常约 300px；含审核说明约 340px |
| 草案四块整卡 | 每块约 280–320px |
| 解释员复盘整卡 | 每块约 210–240px |
| 执行回执 | 每张约 180px |

响应式规则：

- 所有 Grid/Flex 直接子元素补齐 `min-width: 0; max-width: 100%`。
- 520px 以下模块头部改为单列，徽标不得挤压正文。
- 桌面草案为 2×2，移动端单列。
- 审核员操作台桌面使用两列；长 textarea 和提示跨满两列；520px 以下单列。
- 正常正文不滚动；达到该模块异常阈值时使用 `ExpandableText` 或全文弹层。只有结构化超长列表才允许在明确正文区滚动，模块标题和边界提示保持固定。

## 9. 审核工作台

外层 Grid 和冻结索引必须使用 `align-items: start / align-self: start`，避免短卡被长内容强制拉高。

六张审核卡采用 12 列业务布局，并按最窄桌面列宽重新提高固定高度：

```text
双方主张 6列 | 核心争点 6列         高 420px
证据矩阵 7列 | 裁决草案 5列         高 520px
执行方案 7列 | 风险提示 5列         高 280px
```

这里的正常容量指 UI 结构化摘要，不是 ReviewPacket 原始全文：

- 双方主张：每方 80–150 字摘要；完整原文通过详情入口查看。
- 核心争点：3–5 项，每项 30–60 字。
- 证据矩阵：3–6 个争议事实。
- 裁决草案：250–400 字摘要；完整结构化草案从详情区查看。
- 执行动作：1–4 项。
- 风险提示：3–8 项。

超过容量时只让卡片正文区滚动。641–880px 时冻结索引置顶并使用 2×2；≤640px 时长内容卡跨满一列。

倒计时使用两行 Grid：“审核包有效期 + 时间”并列，服务端状态独占第二行。

## 10. 传票信箱

- 桌面宽 420px。
- 移动端宽 100dvw。
- 高度 100dvh。
- 使用 `Teleport to="body"`，消除顶栏定位上下文造成的偏移。

结构：

```text
固定 Header：标题、全部已读、44×44 关闭按钮
可滚动通知列表
```

抽屉本身不滚动。打开时锁定 body；点击通知时先关闭抽屉再导航。

通知卡按摘要设计：标题最多两行、正文最多三行、`min-height: 156px`，正常约 156–164px。完整内容进入对应房间查看。

## 11. 数字人管理页

“推荐”标签从绝对定位改为卡片标题 Grid 的正常内容。身份、标签和流程节点参与布局；文案增长时换行，不能覆盖第一轮节点。

Agent Console 回归数据必须包含：推荐标签 20 字、三个阶段名各 40 字以及 320px 视口，确认标签和节点不重叠。

## 12. 全局响应式与尺寸规则

1. 移除根节点 `min-width: 320px`，避免滚动条占宽后强制产生 14–15px 横向溢出。
2. 房间单双列优先依据容器真实宽度，而不是仅依据 viewport。
3. 断点前后都必须保证外框尺寸稳定、操作可达、文档无横向滚动。
4. 视口高度小于固定卡片时，允许页面纵向滚动，不压缩卡片内部固定区域。
5. 状态码和时间可 `nowrap`；普通说明、标题和按钮文案必须允许合理换行。

## 13. 错误处理与可访问性

- 服务错误使用 toast、对话框或不参与主体 Grid 高度计算的提示层。
- toast/对话框在移动端统一 `max-width: calc(100dvw - 32px)`，错误层自身不得制造横向溢出。
- 展开文本使用真实按钮、`aria-expanded` 和可见焦点。
- 侧边证据抽屉支持 Escape 关闭、焦点约束和关闭后焦点回到入口。
- 移动端关闭按钮和关键操作触控区域不小于 44×44px。
- 内部滚动区设置 `overscroll-behavior: contain` 和稳定滚动条槽位。
- 不以颜色单独表达状态。

## 14. 验收与回归测试

### 14.1 真实浏览器断点

- 庭审：1260/1259、1221/1220、1181/1180、681/680。
- 房间：1121/1120、1061/1060、981/980、621/620、581/580。
- 全局与文档：521/520、461/460、390、320。
- 代表性短屏：1024×600、390×844、320×568。

### 14.2 数据容量

- 1000 字普通聊天消息，以及 1499/1500/2000 字异常展开边界。
- 200 字无空格 URL/文件名/案件标识。
- 150 字摘要、40 字诉求、40 字回应、160 字原始陈述、4 条核验事项。
- 300 字摘要、500 字原始陈述、10 条核验事项。
- 50 条庭审消息、100 张证据卡、100 条通知。
- 约 14k 字符 ReviewPacket。
- Agent Console：20 字推荐标签、三个 40 字阶段名。

### 14.3 强制断言

- `documentElement.scrollWidth <= clientWidth`。
- 固定外框尺寸符合各页合同。
- 只有设计指定的滚动区满足 `scrollHeight > clientHeight`。
- 标题、状态、输入区和操作区始终在外框边界内。
- 正常业务容量下不出现多余展开按钮或小卡内部滚动条。
- 异常长文本不会被静默裁切。
- 窄屏证据抽屉、通知抽屉和全文弹层均可键盘关闭。

## 15. 实施边界与推荐顺序

本设计不改变业务流程、API、房间状态机或消息合同。

推荐实施顺序：

1. 全局宽度规则、共享布局合同和浏览器布局测试基础。
2. 接待室与证据室固定双卡。
3. 庭审画布、消息区和证据抽屉。
4. 总览页。
5. Outcome。
6. Review Workbench。
7. Mailbox 与 Agent Console。
8. 全断点、长文本和多角色端到端回归。

每一步独立验证并提交，避免一次性大范围样式变更导致回归来源不明。
