# 前端长文本与固定窗口布局审计（2026-07-10）

## 1. 审计目标

本轮仅扫描和记录，不修改前端 UI、业务逻辑或样式。

重点验证：

- 卡片内文字超过预估长度时，是否被不可恢复地裁切。
- 固定高度窗口在窄屏、短屏和断点附近是否发生压扁、重叠或操作不可达。
- 长中文、长编号、长文件名、大量列表项是否引起页面级横向滚动。
- 设计上允许的局部滚动、折叠和省略是否仍能保证关键信息可访问。

## 2. 范围与方法

### 2.1 实际生产路由

本轮覆盖 `frontend/src/router/index.js` 中的 8 个实际页面路由：

1. `/disputes`
2. `/disputes/:caseId/intake`
3. `/disputes/:caseId/evidence`
4. `/disputes/:caseId/hearing`
5. `/disputes/:caseId/outcome`
6. `/reviews`
7. `/reviews/:reviewId`
8. `/agents`

另检查全局传票信箱抽屉。旧 `router.js` 中未被 `main.js` 使用的 legacy 页面不计入本轮真实页面覆盖。

### 2.2 浏览器与数据

- 前端：`http://127.0.0.1:5173` / `http://localhost:5173`
- Java API：`http://127.0.0.1:8080`
- 使用本地真实案件、真实庭审消息、真实审核包和真实通知，不使用 jsdom 作为布局结论依据。
- 主要压力视口：`320×568`、`390×844`、`1024×600`，并检查 `1180`、`1020`、`980`、`680`、`620`、`580` 等断点前后。
- 当前 Chrome/DPI 环境中 `1024px` 目标宽度会落到实际 `1023/1025px`，因此以断点两侧双向测量确认；`390×844` 目标实测高度为 `843px`。
- 测量指标：`clientWidth/clientHeight`、`scrollWidth/scrollHeight`、元素 bounding box、页面级横向溢出、关键按钮是否可达。
- 结合静态 CSS 扫描识别固定高度、`overflow: hidden`、`line-clamp`、绝对定位和断点继承风险。

### 2.3 严重级别

- **P0**：核心内容或操作在常见窄屏下不可见、严重重叠，页面主流程无法可靠使用。
- **P1**：出现页面级横向滚动、明显裁切或模块覆盖，仍可绕行但体验和信息完整性显著受损。
- **P2**：当前数据下轻微拥挤或覆盖；文案再增长后容易放大。
- **风险**：静态结构高风险，但当前真实数据不足以确认视觉错乱。

## 6. 现有自动化覆盖缺口

- 当前 Vitest 使用 jsdom，没有 Playwright/Cypress 或其他真实浏览器布局测试。
- 现有“固定布局”“长文本可增长”等测试主要是：
  - 挂载后检查 DOM/class 是否存在；或
  - 读取 `.vue` 源码并断言 CSS 字符串。
- 没有测试真实 `scrollHeight/clientHeight`、元素 bounding box、断点前后 resize、页面横向溢出或关键按钮可达性。

建议后续整改完成后新增真实浏览器布局回归，至少覆盖：

- `documentElement.scrollWidth <= clientWidth`。
- 明确滚动区才允许 `scrollHeight > clientHeight`。
- 固定/粘性区域不遮挡正文与操作按钮。
- 只有设计上明确允许截断、且有 title/展开能力的文本才允许裁切。
- `1181/1180px`、`1021/1020px`、`981/980px`、`681/680px`、`621/620px`、`581/580px`、`521/520px`、`461/460px` 等断点前后各跑一次。
