# Agent 全链路排查与修复报告

## 排查结论

本轮发现的“模型对话 Agent 功能不可用”主要不是前端按钮缺失，而是两类链路问题叠加：

1. 本地 Java 使用 Docker 内网 Agent 地址，导致 Java → Python Agent 调用降级。
2. 本地 Java 未启动 Temporal Worker，导致第三轮庭审信号无人消费，不能生成最终草案和审核任务。

修复后，即使 Python Agent 的底层模型返回 503，系统也能按最终轮次规则生成可审核的确定性草案，并继续进入人工终审。

## 当前 Agent 行为

| Agent/数字人 | 页面 | 验收结果 |
|---|---|---|
| 争议接待官 | 接待室 | 可创建并受理争议，输出接待卷宗；模型不可用时标记 `agent_degraded=true` |
| 证据书记官 | 证据室 | 可接收双方房间消息，证据室倒计时和双方完成状态有效 |
| AI 法官 | 小法庭 | 三轮后生成确定性非最终草案；模型不可用时 fallback 为可审核方案 |
| AI 评审团 | 小法庭/终局 | 当前高风险案件在最终方案后进入审核关注点，后续可继续增强评分展示 |
| 审核解释官 | 平台终审 | 可读取冻结 ReviewPacket，展示事实、证据、规则、草案和补救动作 |

## 修复后的关键保障

- 最终轮次不再允许继续“补交内容”作为终局输出。
- Agent 不可用不等于流程中断：最终轮次会根据卷宗关键词兜底到补发、退款、换货等确定方向。
- 人类终审仍是最终生效点，AI 仅提供非最终建议。
